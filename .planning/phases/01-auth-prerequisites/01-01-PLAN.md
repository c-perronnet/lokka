<task type="auto">
  <name>Task 1: Create Defender auth verification script</name>
  <files>scripts/verify-defender-auth.ts, scripts/tsconfig.json</files>
  <action>
Create a standalone TypeScript verification script at `scripts/verify-defender-auth.ts` that performs three sequential checks:

**Step 1 — Token acquisition:**
- Use `ClientSecretCredential` from `@azure/identity` (already installed in `src/mcp/`)
- Read `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET` from environment variables
- Call `credential.getToken("https://api.securitycenter.microsoft.com/.default")`
- CRITICAL: The scope MUST be `https://api.securitycenter.microsoft.com/.default` — NOT `https://api.security.microsoft.com/.default` (different strings; the latter causes silent 403)
- Print success or failure with the error message

**Step 2 — JWT claims verification:**
- Use `jsonwebtoken` (already installed) to decode the token WITHOUT signature verification (`jwt.decode()`, not `jwt.verify()`)
- Check `aud` claim equals `https://api.securitycenter.microsoft.com` (no trailing slash)
- Check `roles` array includes `Machine.Read.All` OR `Machine.ReadWrite.All`
- Print the actual `aud` and `roles` values for human review
- If `Machine.Read.All` is not present but `Machine.ReadWrite.All` is, print a note: "Using Machine.ReadWrite.All (Machine.Read.All not available as separate permission)"

**Step 3 — EU endpoint API call:**
- HTTP GET `https://eu.api.security.microsoft.com/api/machines?$top=1` with `Authorization: Bearer {token}`
- Use global `fetch` (Node 18+ built-in, no polyfill needed for a standalone script)
- Print HTTP status code
- On 200: print "EU endpoint reachable" and show the first few lines of the response body (truncated to avoid dumping large payloads)
- On non-200: print the full response body for debugging

**Output format:** Each step prints a clear PASS/FAIL line with relevant details. The script exits with code 0 if all three checks pass, code 1 if any fails.

**Script infrastructure:**
- Create `scripts/tsconfig.json` extending the parent config but targeting the scripts directory:
  ```json
  {
    "compilerOptions": {
      "target": "ES2022",
      "module": "Node16",
      "moduleResolution": "Node16",
      "outDir": "./build",
      "rootDir": ".",
      "strict": true,
      "esModuleInterop": true,
      "skipLibCheck": true
    }
  }
  ```
- The script should import from the existing `@azure/identity` and `jsonwebtoken` packages (resolved from `src/mcp/node_modules/` via Node module resolution — no separate install needed).
- Add a shebang-style comment at the top explaining how to run it:
  ```
  // Run: cd src/mcp && npx tsc --project ../../scripts/tsconfig.json && node ../../scripts/build/verify-defender-auth.js
  // Requires: AZURE_TENANT_ID, AZURE_CLIENT_ID, AZURE_CLIENT_SECRET environment variables
  ```

Do NOT import from Lokka source files (auth.ts, etc.) — this script must be self-contained so it works even if the Lokka build is broken.

**Part A — Azure Portal check (do this first if not already done):**
1. Go to Azure Portal > App registrations > find the Lokka app
2. Click "API permissions"
3. Click "Add a permission" > "APIs my organization uses" > type "WindowsDefenderATP"
4. Select "Application permissions" > look for `Machine.Read.All` (preferred) or `Machine.ReadWrite.All`
5. Add the permission
6. Click "Grant admin consent for [your tenant]"
7. Verify a green checkmark appears next to the permission

If "WindowsDefenderATP" does not appear in the search, the tenant may lack a Defender for Endpoint license — this is a blocker.

**Part B — Run the verification script:**
```bash
cd src/mcp

export AZURE_CLIENT_ID="your-client-id"
export AZURE_CLIENT_SECRET="your-client-secret"
npx tsc --project ../../scripts/tsconfig.json
node ../../scripts/build/verify-defender-auth.js
```

**Expected output:** All three steps show PASS:
- Step 1: Token acquired successfully
- Step 2: `aud` = `https://api.securitycenter.microsoft.com`, `roles` includes Machine.Read.All or Machine.ReadWrite.All

**If Step 2 fails (wrong aud):** The scope constant is wrong — report the actual `aud` value.
**If Step 2 fails (missing roles):** Admin consent not granted — redo Part A step 6.
**If Step 3 fails (403):** Likely wrong scope or missing permissions — check Step 2 output first.
**If Step 3 fails (network error):** Firewall or DNS issue with `eu.api.security.microsoft.com`.
